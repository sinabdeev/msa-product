# 📱 ТЗ: Android-приложение "Product Status History Dashboard"

## 1. Цель
Разработать нативное Android-приложение для визуализации аналитики статусов продуктов. Приложение должно потреблять данные с существующего бэкенда (аналогично веб-версии) и отображать их в виде интерактивного дашборда с графиками.

## 2. Технологический стек
*   **Язык:** Kotlin
*   **UI Toolkit:** Jetpack Compose (Material 3)
*   **Архитектура:** MVVM + Clean Architecture
*   **Сеть:** Retrofit + OkHttp + Kotlin Serialization (или Moshi)
*   **DI:** Hilt
*   **Асинхронность:** Coroutines + Flow
*   **Графики:** Библиотека **Vico** (рекомендуется для Compose) или MPAndroidChart
*   **Навигация:** Navigation Compose

## 3. Функциональные требования

### 3.1. Экран Дашборда (Главный)
Экран представляет собой адаптивную сетку карточек (`LazyVerticalGrid`).
*   **Адаптивность:**
    *   Смартфон (портрет): 1 колонка.
    *   Смартфон (ландшафт) / Планшет: 2-3 колонки (как на веб-скрине).
*   **Обновление данных:** Pull-to-refresh (свайп вниз) для принудительного обновления всех графиков.
*   **Состояния загрузки:**
    *   *Loading:* Шимер-эффект (placeholder) внутри карточек.
    *   *Error:* Сообщение об ошибке с кнопкой "Повторить" внутри конкретной карточки или全局льно.
    *   *Empty:* Заглушка, если данных нет.

### 3.2. Виджеты (Карточки графиков)
Необходимо реализовать следующие компоненты на основе веб-версии:

| № | Название виджета | Тип графика | Данные (примерные поля API) | Примечания |
| :--- | :--- | :--- | :--- | :--- |
| 1 | Переходы по статусам | Vertical Bar Chart | `status`, `count` | Ось X: статусы (ARCHIVED, PENDING...), Ось Y: кол-во. Цвета фиксированные по статусам. |
| 2 | Среднее время обработки | Single Bar / Gauge | `avg_time` | Один столбец или прогресс-бар. |
| 3 | Топ продуктов | Horizontal Bar Chart | `product_id`, `count` | Показывать топ-10. ID могут быть длинными — предусмотреть обрезку текста (ellipsis). |
| 4 | Причины переходов | Vertical Bar Chart | `reason`, `count` | Обычно один доминирующий бар ("Batch processing"). |
| 5 | Доля статусов | Pie Chart | `status`, `percentage` | Легенда снизу или сбоку. Цвета должны совпадать с виджетом №1. |
| 6 | Доля причин (donut) | Donut Chart | `reason`, `percentage` | Аналогично Pie Chart, но с отверстием в центре. |
| 7 | Откуда переходят | Pie Chart | `source_status`, `count` | Показывает распределение начальных статусов. |
| 8 | Активность по часам | Radial Bar / Pie | `hour`, `activity_level` | Если данных мало (как на скрине "23:00"), использовать круговую диаграмму. Если много — Bar Chart по 24 часам. |
| 9 | Активность пользователей | *TBD* | - | На скрине пусто. Уточнить у аналитиков/бэкенда, нужны ли эти данные. |

### 3.3. Цветовая схема (Design System)
Цвета графиков должны быть вынесены в ресурсы (`colors.kt` или тему), чтобы соответствовать брендингу:
*   🟣 ARCHIVED / DRAFT: `#8E86D6` (Фиолетовый)

*   🟡 REJECTED (light): `#FAD7A0` (Желтый)

## 5. Критерии приемки (DoD)
1.  Приложение собирается и запускается на Android 8.0+ (API 26).
2.  Все графики из веб-версии отображаются корректно.
3.  Цвета графиков соответствуют брендбуку/веб-версии.
4.  При повороте экрана верстка адаптируется (сетка перестраивается).
5.  При отсутствии интернета показывается понятная ошибка, а не краш.
6.  Данные обновляются при свайпе вниз.

#  ТЗ: Android Dashboard (Native Kotlin) — Финальная версия

## 1. Технологический стек
*   **Язык:** Kotlin
*   **UI:** Jetpack Compose + Material 3
*   **Архитектура:** MVVM + Repository Pattern
*   **Сеть:** Retrofit + OkHttp + Kotlin Serialization
*   **DI:** Hilt
*   **Графики:** Vico (рекомендуется для Compose) или MPAndroidChart
*   **Логирование:** Timber (аналог `logger` из веба)
*   **Локальное хранилище настроек:** DataStore Preferences

## 2. Интеграция с API (КОНТРАКТ УТОЧНЕН)

### 2.1. Эндпоинт
*   **URL:** `GET /api/v1/status-history?limit={initialLimit}`
*   **Метод:** GET
*   **Параметры:**
    *   `limit` (Int) — количество записей для загрузки. Значение берется из настроек приложения (`initialLimit`).

### 2.2. Формат ответа
Бэкенд возвращает обертку `ApiResponse<T>`:
```json
{
  "success": true,
  "data": [ /* массив StatusHistoryRecord */ ],
  "message": null
}
```
**Важно:** В Android нужно создать generic-класс `ApiResponse<T>` и парсить поле `success`. Если `false` — выбрасывать `ApiError` с сообщением из поля `message`.

### 2.3. Модель данных (StatusHistoryRecord)
Необходимо получить точную структуру `StatusHistoryRecord` из файла `../types/data` веб-проекта.
**Примерная структура (уточнить!):**
```kotlin
data class StatusHistoryRecord(
    val id: String,
    val productId: String,
    val fromStatus: String?,
    val toStatus: String,
    val reason: String?,
    val timestamp: String, // ISO 8601
    val userId: String?,
    val processingTimeMs: Long?
)
```

### 2.4. Агрегация данных на клиенте
**Критически важно:** Бэкенд отдает **сырые записи** (`status-history`), а не агрегированные данные для графиков.
**Задача Android:** Реализовать локальную агрегацию в Repository/UseCase слое.
*   `aggregatedData` формируется из списка `StatusHistoryRecord` прямо в приложении.
*   Это позволяет избежать создания 9 отдельных эндпоинтов на бэкенде.

**Пример логики агрегации:**
```kotlin
fun List<StatusHistoryRecord>.toAggregatedData(): AggregatedDashboard {
    return AggregatedDashboard(
        statusTransitions = groupBy { it.toStatus }.mapValues { it.value.size },
        avgProcessingTime = mapNotNull { it.processingTimeMs }.average(),
        topProducts = groupBy { it.productId }
            .mapValues { it.value.size }
            .toList()
            .sortedByDescending { it.second }
            .take(10),
        transitionReasons = groupBy { it.reason ?: "Unknown" }.mapValues { it.value.size },
        // ... остальные метрики аналогично
    )
}
```

## 3. UI Экраны

### 3.1. Навигация
Использовать `NavigationBar` (Bottom Navigation) с 3 вкладками:
1.  **Charts** (Главная) — иконка 📊
2.  **Data** (Таблица) — иконка 📋
3.  **Settings** (Настройки) — иконка ⚙️

### 3.2. Экран Charts
*   **Layout:** `LazyVerticalGrid(columns = GridCells.Adaptive(300.dp))`
*   **Карточки:** 9 виджетов (см. предыдущее ТЗ). Данные берутся из локально агрегированного `AggregatedDashboard`.
*   **Обновление:** `PullRefreshIndicator` → перезапрос `/api/v1/status-history?limit=...` → ре-агрегация → обновление UI.
*   **Состояния:**
    *   *Loading:* Shimmer внутри каждой карточки.
    *   *Error:* Snackbar + кнопка Retry.
    *   *Empty:* Заглушка "Нет данных".

### 3.3. Экран Data
*   **Компонент:** `LazyColumn` с таблицей сырых записей.
*   **Данные:** Список `StatusHistoryRecord` (без агрегации).
*   **Пагинация:** Если записей > 100, реализовать подгрузку через `Paging 3 Library` или бесконечный скролл с увеличением `limit`.

### 3.4. Экран Settings
*   **Поля:**
    *   `Initial Limit` (Int, default: 50) — сохраняется в DataStore.
    *   `Poll Interval` (Int, seconds, default: 30) — интервал автообновления (опционально).
*   **Кнопка:** "Reset to Defaults".
*   **Сохранение:** Все настройки сохраняются в `DataStore Preferences` и применяются немедленно.

## 4. Обработка ошибок
*   **HTTP 4xx/5xx:** Показать сообщение из поля `message` ответа API.
*   **Network Error:** Показать "Проверьте подключение к интернету".
*   **Parse Error:** Логировать в Timber, показать "Ошибка формата данных".
