# Реализация возможности комбинирования путей (динамические переходы)

## Цель
Обеспечить возможность продукту двигаться по произвольной последовательности допустимых переходов, комбинируя пути на ходу, без жесткой привязки к заранее определенным маршрутам.

## Концепция
- **Путь** — это последовательность статусов, которую продукт проходит от начального до конечного состояния.
- **Комбинирование путей** — возможность начать с одного пути, затем переключиться на другой в зависимости от бизнес-событий.
- **Динамические переходы** — каждый переход проверяется только на соответствие матрице допустимых переходов, без учета предыдущей истории (кроме некоторых бизнес-ограничений).

## Пример комбинирования
1. Продукт начинает по пути 1: `DRAFT → PENDING_REVIEW → REVIEWED`
2. Затем отклоняется (переход на путь 2): `REVIEWED → REJECTED`
3. Исправляется и возвращается в черновик (новый переход): `REJECTED → PENDING_REVIEW` (но по матрице это допустимо только в `PENDING_REVIEW` или `ARCHIVED`, поэтому нужно `REJECTED → PENDING_REVIEW`)
4. Снова проходит проверку: `PENDING_REVIEW → REVIEWED`
5. Утверждается: `REVIEWED → APPROVED`
6. И т.д.

## Реализация

### 1. Гибкая матрица переходов
Матрица переходов уже позволяет комбинирование, если она построена правильно. Например:
- `REJECTED` может перейти в `PENDING_REVIEW` (исправление)
- `PENDING_REVIEW` может перейти в `DRAFT` (возврат для правок)
- `ARCHIVED` может перейти в `ACTIVE` (восстановление)

Таким образом, продукт может "прыгать" между статусами, если переход разрешен.

### 2. Отсутствие привязки к предопределенным путям
Система не должна хранить или проверять, соответствует ли текущая последовательность какому-либо заранее заданному пути. Вместо этого каждый отдельный переход валидируется независимо.

### 3. История переходов как источник фактического пути
Фактический путь продукта восстанавливается из истории переходов (`ProductStatusHistoryService`). Это позволяет:
- Анализировать, по какому пути фактически прошел продукт
- Выявлять неэффективные циклы (например, многократные отклонения)
- Строить отчеты о времени пребывания в каждом статусе

### 4. Бизнес-правила для ограничения комбинирования
Некоторые комбинации могут быть нежелательными с бизнес-точки зрения, даже если они разрешены матрицей. Например:
- Нельзя переходить из `ARCHIVED` в `ACTIVE` более 3 раз
- Нельзя возвращаться в `DRAFT` после того как продукт был `APPROVED`

Эти правила должны быть реализованы как дополнительные валидаторы в `ProductStatusTransitionService`.

## Дополнительные валидаторы

Создадим интерфейс `ProductStatusTransitionRule`, который позволяет добавлять бизнес-правила поверх базовой матрицы.

```kotlin
package ru.example.product.processing.service.status.rules

import ru.example.product.processing.domain.ProductEntity
import ru.example.product.processing.domain.ProductStatus

interface ProductStatusTransitionRule {
    fun validate(current: ProductStatus, target: ProductStatus, product: ProductEntity)
}
```

Примеры правил:
1. **NoBackToDraftAfterApprovalRule** — запрещает переход в `DRAFT` после того как продукт был `APPROVED`
2. **MaxArchivedRestorationsRule** — ограничивает количество восстановлений из `ARCHIVED`
3. **MinimumTimeInStatusRule** — требует минимального времени пребывания в статусе перед переходом

## Реализация в ProductStatusTransitionService

`ProductStatusTransitionService` должен:
1. Проверить базовую допустимость перехода через `ProductStatusTransitionValidator`
2. Применить все зарегистрированные правила `ProductStatusTransitionRule`
3. Если хотя бы одно правило нарушено — выбросить `InvalidStatusTransitionException`

## Конфигурация правил

Правила могут быть настроены как Spring бины и инжектированы в сервис через список:

```kotlin
@Service
class ProductStatusTransitionServiceImpl(
    private val validator: ProductStatusTransitionValidator,
    private val rules: List<ProductStatusTransitionRule> = emptyList()
) : ProductStatusTransitionService {
    // ...
}
```

## Пример динамического пути

Рассмотрим сценарий:
1. Продукт создан (`DRAFT`)
2. Отправлен на проверку (`PENDING_REVIEW`)
3. Проверен (`REVIEWED`)
4. Отклонен (`REJECTED`)
5. Исправлен и отправлен повторно (`PENDING_REVIEW`)
6. Проверен (`REVIEWED`)
7. Утвержден (`APPROVED`)
8. Активирован (`ACTIVE`)
9. Снят с продажи (`ARCHIVED`)
10. Восстановлен (`ACTIVE`)
11. Обработан (`PROCESSED`)
12. Отправлен (`SHIPPED`)
13. Заархивирован (`ARCHIVED`)

Этот путь комбинирует элементы путей 1, 2, 6, 10 из документа `product_status_workflow.md`.

## Тестирование комбинирования

Необходимо написать интеграционные тесты, которые:
1. Создают продукт
2. Выполняют серию переходов, имитирующих комбинирование путей
3. Проверяют, что конечный статус корректен
4. Проверяют, что история переходов содержит все промежуточные статусы

## План реализации

### Этап 1: Базовая динамика (неделя 1)
- Реализовать `ProductStatusTransitionValidator` с матрицей переходов
- Реализовать `ProductStatusTransitionService` с базовой валидацией
- Написать тесты для последовательных переходов

### Этап 2: Бизнес-правила (неделя 2)
- Создать интерфейс `ProductStatusTransitionRule`
- Реализовать 2-3 ключевых правила (например, запрет возврата в черновик после утверждения)
- Интегрировать правила в сервис переходов

### Этап 3: История и анализ (неделя 3)
- Реализовать `ProductStatusHistoryService` с записью всех переходов
- Добавить метод для получения фактического пути продукта
- Создать отчет о времени в статусах

### Этап 4: Мониторинг и оптимизация (неделя 4)
- Добавить метрики переходов (сколько раз продукт менял статус)
- Выявлять циклы (например, продукт многократно отклоняется)
- Предоставить API для получения возможных следующих шагов

## Заключение

Комбинирование путей достигается за счет:
1. Гибкой матрицы переходов, разрешающей множество комбинаций
2. Отсутствия жесткой привязки к предопределенным маршрутам
3. Возможности добавления бизнес-правил для ограничения нежелательных комбинаций
4. Ведения истории переходов для анализа фактического пути

Это позволяет продукту адаптироваться к изменяющимся бизнес-условиям и проходить индивидуальные жизненные циклы.
